## Root Cause Found Late

The most important discovery:

The project's `Start date` and `Target date` fields look like normal ProjectV2 fields in the project metadata:

- `Start date`: `PVTF_lADODNQhvs4BY49yzhT7iEY`
- `Target date`: `PVTF_lADODNQhvs4BY49yzhT7iEc`

But GitHub rejects writes through `updateProjectV2ItemFieldValue` for these fields:

```text
Issue field values cannot be updated using the updateProjectV2ItemFieldValue mutation,
they must be updated using the updateIssueFieldValue mutation
```

This explains why the app appeared to update locally but GitHub readback still showed no persisted Start Date.

Manual repro command that exposed this:

```bash
gh api graphql \
  -f query='mutation($projectId:ID!,$itemId:ID!,$fieldId:ID!,$date:Date!){ updateProjectV2ItemFieldValue(input:{projectId:$projectId,itemId:$itemId,fieldId:$fieldId,value:{date:$date}}){ projectV2Item { id } } }' \
  -f projectId=PVT_kwDODNQhvs4BY49y \
  -f itemId=PVTI_lADODNQhvs4BY49yzgt4kzs \
  -f fieldId=PVTF_lADODNQhvs4BY49yzhT7iEY \
  -F date=2026-05-27
```

The response was a validation error telling us to use `updateIssueFieldValue`.

Problem:

`updateIssueFieldValue` does not accept the ProjectV2 field ID. A manual mutation using:

```json
{
  "fieldId": "PVTF_lADODNQhvs4BY49yzhT7iEY",
  "dateValue": "2026-05-27"
}
```

returned:

```text
Issue fields Issue field with id 352028742 not found.
```

So the next agent must discover how to map the ProjectV2 field to the backing issue-field ID.

Useful schema facts already discovered:

```graphql
UpdateIssueFieldValueInput {
  issueId: ID!
  issueField: IssueFieldCreateOrUpdateInput!
}

IssueFieldCreateOrUpdateInput {
  fieldId: ID!
  textValue: String
  dateValue: String
  singleSelectOptionId: ID
  multiSelectOptionIds: [ID!]
  numberValue: Float
  delete: Boolean
}

CreateProjectV2IssueFieldInput {
  projectId: ID!
  issueFieldId: ID!
}
```

`ProjectV2Field` introspection only exposed:

- `createdAt`
- `dataType`
- `databaseId`
- `id`
- `name`
- `project`
- `updatedAt`

It did not expose an obvious backing `issueFieldId`.
