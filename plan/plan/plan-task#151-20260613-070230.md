# Task #151 Plan (Revision 2): Incremental GitHub Sync and Viewport Preservation

Issue: https://github.com/mypals-ml/glidepace-vibe/issues/151
Supersedes: `plan/plan/plan-task#151-20260612-221852.md` (Revision 1)

## Goal

Avoid making the opened project look like it fully refreshed when GitHub webhook
events arrive.

Required behavior from the issue:

- If GitHub sends a single element change event, update and change that element
  only.
- If GitHub sends a full refresh event, update all data and keep the first
  available currently-displayed task item at the viewport top as much as
  possible.
- Handle the cases where currently-displayed task items were moved or deleted.
- Deliverable for this round: a plan document for review (this file). No
  implementation is performed in this round.

## Remaining Work Plan (Phase 2)

Phase 2 is intentionally small and should only proceed after this plan is
reviewed. Ordered by value:

### 1. Component/Integration Tests (highest value, lowest risk)

Add tests that exercise the wiring, not just the pure helpers:

- `ProjectDashboard` (or a focused harness around its anchor effect):
  - after a simulated background snapshot apply, `sidebarRef.scrollTop` and
    `timelineRef.scrollTop` are restored to the anchored task's position;
  - when the anchored task was deleted, the next surviving candidate is used;
  - stale anchors (older than `VIEWPORT_ANCHOR_MAX_AGE_MS`) are discarded.
- `useDashboardSync` behavior tests with a mocked Supabase channel:
  - duplicate broadcasts on project/repo/owner channels → exactly one fetch;
  - `refresh_task` for a known item → `fetchSingleProjectItem`, no full fetch;
  - `refresh_task` for an unknown item → one background full refresh;
  - full refresh requested while one is in flight → coalesced trailing run.
- Selected-task cleanup: snapshot that removes the currently selected task
  closes/cleans details state predictably (add the missing behavior if the
  test reveals a gap).

### 2. Render Granularity (only if profiling justifies it)

Current state: `SortableTaskRow` is memoized (`TaskSidebarRows.tsx`), and
reconciliation preserves object identity, so sidebar rows already skip
rerenders. `GanttChart.tsx` (697 lines) renders task/group bars inline with
`useMemo`d derived data but no per-row memoized components.

- Profile with React DevTools in mock mode (`npm run dev:test`) while sending
  simulated single-item and full-refresh events.
- If Gantt rendering is visibly expensive: extract memoized `GanttTaskRow` /
  `GanttGroupRow` keyed by stable IDs (follows the `TaskSidebarRows` pattern;
  also aligns with the file-size guidance from task #145).
- Only if context-wide rerenders remain the bottleneck: introduce a
  no-dependency selector store via `useSyncExternalStore`
  (`useDashboardItemIds()`, `useTaskByStableId(id)`), keeping
  `DashboardContext` for commands and UI state. Do not add Zustand/Redux.

### 3. Manual Verification Matrix

Run `npm run dev:test`, open a mock project with enough rows to scroll, then:

| Scenario | Expected |
| --- | --- |
| Issue title/status/assignee edit (single-item event) | Only the affected row/bar updates; no spinner; scroll position untouched |
| Project item field edit | Same as above; dependency cascade may update successor bars |
| External reorder | Background full refresh; first surviving visible task stays at its offset from viewport top |
| Item deletion/archive of the top visible task | Next visible surviving task becomes the anchor |
| Full sync fallback (`sync` event) | All data refreshes; no UI blanking; anchor restored |
| Background refresh network failure | Stale list stays visible; error toast shown; no error panel |
| Duplicate broadcasts (project+repo+owner) | One network refresh in logs |

## Risks and Edge Cases (carried forward, still applicable)

- `issues`-sourced webhooks never carry `projectId`; they must keep being
  treated as "possibly ours" and resolved via `contentId`.
- GitHub API eventual consistency can return stale data shortly after a
  webhook; the webhook-side delay and local-edit protection must be retained.
- A single dependency edit can legitimately change successor display dates, so
  "single element" does not always mean one rendered row changes.
- A refresh can move the anchor task into a collapsed group; the anchor
  resolver already falls through to the next candidate, and the unit tests
  cover it.
- Phase 2's external-store refactor is deliberately gated behind profiling
  evidence: it is the largest-risk item and the current memoized structure may
  already meet the acceptance criteria.
