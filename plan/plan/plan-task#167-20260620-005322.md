## Root Cause Analysis (from code inspection via reads/greps)
Two related symptoms:
1. **Cannot change via app UI**: The <input type="date"> for startDate in TaskDetailsContent.tsx calls updateTaskDates, which only writes the field when fieldId is resolved. projectFieldIds.startDate is populated *only* inside mapProjectItemToTask *if* the value node for the start field happens to be present in that item's fieldValues response (i.e. only for tasks that already have a Start Date value). dateSettings.startDateFieldId is per-project from localStorage and auto-detected by keyword, but for first-time or certain projects it may not be in scope for the task obj. Result: fieldId falsy -> no addSet for start -> changes.length===0 -> anySuccess=false -> optimistic revert. No write to GH.
2. **GH direct edit not reflected (stale/old value shown)**: External changes to Start Date field arrive via webhook 'edited' -> 'refresh_task' -> fetchSingleProjectItem -> map (gets actualStartDate) -> reconcileSingleTask (mergeFetched + cascadeTaskDates). merge protects using localUpdateTimestamp (30s window from prior edits). More critically, cascadeAllTasks (full refreshes) and even some paths can inject tempStartDate (via getEffective preferring temp) shadowing the real startDate from fetched. getStartDateForCal prefers temp. Thus GH value not visible in UI/cals even after refresh.
Related: projectFieldIds only partially populated; cascade/merge not strongly preferring real GH startDate over temps when present; no guarantee of fieldId for writes on "unset" start date tasks.

## Implementation Plan
1. **Always populate configured field IDs in mapper** (ensures update can always resolve startDateFieldId etc via task fallback or direct):
   - In mapProjectItemToTask (githubTaskMapper.ts), after the per-value forEach, *unconditionally* set:
     if (dateSettings?.startDateFieldId) projectFieldIds.startDate = dateSettings.startDateFieldId;
     else if (startDateField?.field?.id) ...
     - Same pattern for targetDateFieldId, estimateFieldId, estimateUnitFieldId, successor, predecessor, groupPathFieldId.
   - This makes every Task carry the field IDs from settings (populated at project load time). Fixes first-time set for Start Date.

2. **Make start date write robust in mutations**:
   - In updateTaskDates, change the start write guard:
     } else if (normalizedStartDate) {
       const fieldId = dateSettings.startDateFieldId || task.projectFieldIds?.startDate;
       addSet(fieldId, { date: formatToGitHubDate(normalizedStartDate) });
     }
   - Use normalized (after clear handling) for the truthy check.
   - When start/target involved in the call, skip the immediate fetchSingleProjectItem (pass or conditional !skip && !dateChange) to avoid potential eventual-consistency stale read clobbering the just-written optimistic value. Webhook 'refresh_task' will reconcile later.

3. **Fix GH external changes not reflected**:
   - In mergeFetchedTaskWithLocalState (taskMergeUtils.ts): *always* adopt fetched values for date-related fields (do not protect startDate/targetDate/tempStart/tempTarget under the 30s local timestamp). Protection still applies to title/body/status/assignees/comments/successors etc.
     - Rationale: our date writes go through optimistic + webhook delay is sufficient; external GH date edits must win immediately on sync. Avoids masking bug.
   - Additionally, in mapper return, explicitly force-clear temps when real values present:
     tempStartDate: actualStartDate ? undefined : partialTask.tempStartDate,
     tempTargetDate: actualTargetDate ? undefined : partialTask.tempTargetDate,
   - This ensures get*ForCal and display prefer real GH values.

4. **Minor hardening** (if needed during review):
   - In TaskDetailsContent start input onChange, ensure we pass the value.
   - No changes to cascade logic (the fixed/Boolean(startDate) already protects manuals unless mode==='auto').

No new fields, no schema change. Pure bugfix + robustness.
