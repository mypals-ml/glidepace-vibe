# Task #151 Plan: Incremental GitHub Sync and Viewport Preservation

Issue: https://github.com/mypals-ml/glidepace-vibe/issues/151

## Goal

Avoid making the opened project look like it fully refreshed when GitHub webhook events arrive.

Required behavior:

- Single item change events update only the affected task row/bar and any dependency-derived rows that genuinely need recalculation.
- Full refresh events reload all project data without blanking the current UI.
- After a full refresh, keep the first still-available task item that was visible before the refresh as close as possible to the viewport top, even if other visible items moved or were deleted.

## Current State

- `api/github-webhook.ts` classifies webhooks into `sync`, `refresh_task`, and `reorder`.
- `src/hooks/useDashboardSync.ts` subscribes to project, repo, and owner Supabase channels. The same webhook can arrive through multiple channels.
- `refresh_task` calls `fetchSingleProjectItem` when an `itemId` is available, or resolves `contentId` from current tasks.
- `sync` and external `reorder` call `fetchProjectTasks`, which sets `isLoadingTasks` to `true`, fetches all pages, then replaces the task array.
- `src/hooks/useDashboardTasks.ts` stores tasks as an ordered array. It already uses `mergeFetchedTaskWithLocalState` to protect recent local edits.
- `TaskSidebar` renders `dashboardItems` and uses memoized `SortableTaskRow`; preserving unchanged task object identities will reduce row rerenders.
- `GanttChart` renders task/group bars inline from `dashboardItems`, so it will still rerender more broadly until row/bar rendering is split later.
- `ProjectDashboard` owns the shared sidebar/timeline scroll refs via `useScrollSync`. Both list and Gantt currently use fixed `72px` row height (task rows and group rows alike), which makes viewport anchoring practical. There is no separate `DashboardLayout` component; `ProjectDashboard.tsx` is the owner of `sidebarRef`/`timelineRef`.
- On fatal GraphQL errors, `fetchProjectTasks` calls `setApiError(...)` and `setTasks([])`; `TaskSidebar` replaces the entire list with an error panel whenever `apiError` is set. Both behaviors must be bypassed in background refresh mode.
- Both `fetchSingleProjectItem` and `fetchProjectTasks` duplicate their merge/cascade logic twice: once as an "AtThisInstant" snapshot for persistence side-effects, once inside the functional `setTasks` updater. The reconciliation utilities should collapse this duplication, not inherit it.
- The same `refresh_task` event arriving on project, repo, and owner channels can today trigger up to three concurrent `fetchSingleProjectItem` calls for the same item.

## Proposed Approach

Use a small in-repo sync/reconciliation layer before adding a new state-management dependency.

The first implementation should keep `DashboardProvider` and the current hook structure, but add explicit data reconciliation, event dedupe, background refresh mode, and viewport anchoring. If profiling still shows expensive broad renders after this, add a selector-based task store with React's built-in `useSyncExternalStore` rather than bringing in Zustand/Redux.

## Design

### 1. Normalize and Dedupe Sync Events

Add `src/lib/githubSyncEvents.ts` with typed helpers:

- `normalizeGithubSyncPayload(rawPayload, channelLabel)` returns a single internal event shape:
  - `kind: 'single_task' | 'full_project' | 'reorder'`
  - `projectId`
  - `itemId`
  - `contentId`
  - `action`
  - `deliveryId` or deterministic fingerprint
  - `receivedAt`
- `isEventForSelectedProject(event, selectedProjectId)` centralizes project filtering.
- `getSyncEventFingerprint(event)` supports short-lived duplicate suppression.

Update `api/github-webhook.ts` to include:

- GitHub delivery header (`x-github-delivery`) when available. This is the robust dedupe key because the handler already reuses one `broadcastPayload` (with a single `timestamp: Date.now()`) across all three channels.
- Source event name and action.
- `projectId`, `itemId`, `contentId`, and `timestamp` for all event kinds where available. Today `sync` broadcasts carry only `{ message, timestamp }`.

Webhook caveats discovered in the current handler:

- `issues` events carry no `projects_v2_item`, so `projectId` is undefined for them; they reach the client only via repo/owner channels. `isEventForSelectedProject` must treat a missing `projectId` as "possibly ours" (proceed with refresh), not filter it out.
- The 1–2 s eventual-consistency delay currently runs only inside the `if (projectId)` block, so `issues` events broadcast with no delay. If the delay is kept, move it before all channel sends.

Fingerprint fallback when `deliveryId` is absent: `event + action + (itemId || contentId || 'project') + timestamp`.

Update `useDashboardSync.ts` to:

- Use one shared handler for project, repo, and owner channels.
- Suppress duplicate event fingerprints for a TTL such as 10 seconds.
- Coalesce full refresh requests per project while a refresh is already in flight.
- Keep the local reorder echo skip behavior from `reorderSyncUtils`.

### 2. Add Task Reconciliation Utilities

Add `src/lib/taskReconciliation.ts` with pure functions:

- `getTaskStableId(task)`:
  - Prefer `itemId`.
  - Fall back to `contentId`.
  - Fall back to `id`.
- `reconcileSingleTask(currentTasks, fetchedTask, options)`:
  - Update an existing task in place (by stable ID).
  - If the task is not present, do not append blindly (current code appends at the array end, which breaks GitHub ordering). Return a `needsFullRefresh` signal so the caller falls back to a background full refresh instead.
  - Preserve unchanged task object references.
  - Use `mergeFetchedTaskWithLocalState` for existing tasks.
  - Run dependency auto-correction and date cascade from the changed task only.
  - Return `{ tasks, changedTaskIds, dependencyCorrections }`.
- `reconcileProjectSnapshot(currentTasks, fetchedTasks, options)`:
  - Apply the full ordered GitHub snapshot.
  - Preserve object identity for tasks whose relevant fields did not change.
  - Preserve recent local edits through `mergeFetchedTaskWithLocalState`.
  - Preserve existing comments when the snapshot did not include comments.
  - Detect added, updated, moved, and removed IDs for debug logs and tests.
  - Run `autoCorrectDependencyFields` and `cascadeAllTasks`.

Design constraint: both fetch functions currently compute merge/cascade twice (an "AtThisInstant" snapshot for persistence plus a functional `setTasks` updater). The reconciliation functions must be pure and cheap enough to run once inside the functional updater, surfacing `corrections`/`changedTaskIds` to the caller via an out-param ref so the duplicated snapshot computation can be deleted rather than ported.

This keeps the existing array API for components while creating a data layer boundary that can later back a normalized store.

### 3. Split Initial Loading From Background Refresh

Change `fetchProjectTasks` into a refresh API that accepts a mode:

```ts
fetchProjectTasks(projectId, token, {
  mode: 'initial' | 'background',
  reason: 'initial_load' | 'manual_sync' | 'webhook_sync' | 'external_reorder' | 'fallback',
  preserveViewport?: boolean,
})
```

Behavior:

- Initial load with no existing tasks can show the current loading UI.
- Background refresh keeps the existing task list and Gantt visible while the network request runs.
- `isLoadingTasks` should only mean "no usable task data yet".
- Add a separate lightweight state such as `isRefreshingTasks` or reuse sync-status UI for background refresh progress.
- Do not clear `tasks` during background refresh failures. Specifically, the fatal-GraphQL-error branch in `fetchProjectTasks` currently calls `setTasks([])` — that must be skipped in background mode.
- Do not set `apiError` in background mode either: `TaskSidebar` swaps the whole list for an error panel when `apiError` is set. Surface background failures via toast or the sync-status indicator instead, leaving the stale-but-usable list in place.
- Keep field progress updates, but avoid replacing the main task area with a spinner when tasks already exist.
- Thread the new options through the plumbing: `DashboardProvider` holds `fetchProjectTasksRef` typed `(id, token) => Promise<void>` and the context/`useDashboardSync` props use the same signature. All of these type signatures need the optional third argument.

### 4. Single Item Event Flow

For `single_task` events:

- If `itemId` exists, call `fetchSingleProjectItem`.
- If only `contentId` exists, find the current task by `contentId` and call `fetchSingleProjectItem(task.itemId)`.
- If the item cannot be resolved, fall back to a background full refresh with viewport preservation.
- If fetching a known `itemId` returns `null` for an archive/delete-like action, remove that item locally if safe, otherwise fall back to a background full refresh.
- Do not set project-level loading state for single item refreshes.
- Do not update fields/project metadata unless the single item response proves a field option mapping changed.
- Add a per-`itemId` in-flight guard in addition to event fingerprint dedupe: today the same `refresh_task` arriving on project, repo, and owner channels triggers up to three concurrent `fetchSingleProjectItem` calls for the same item.

Important caveat: dependency fields can make one item change affect successor dates. The event should still be treated as a single item fetch, but reconciliation may update affected successor task display dates.

### 5. Full Refresh Event Flow

For `sync`, external `reorder`, unknown event fallback, and collection-level changes:

- Fetch all fields/items in the background.
- Reconcile the new snapshot with current tasks.
- Capture the viewport anchor immediately before applying the reconciled snapshot (just before `setTasks`), not before starting the fetch — the fetch can take seconds and the user may keep scrolling during it. Capturing at apply time anchors against what the user actually sees at the moment of the swap.
- Apply the new tasks without blanking the UI.
- Restore the viewport anchor after `dashboardItems` has been recomputed and rendered.

Reorder handling:

- Local reorder echoes should continue to be skipped.
- External reorder still needs a full project snapshot unless GitHub supplies enough ordering context to patch order locally.
- The refresh should be background-only and viewport-preserving.

### 6. Viewport Anchor Layer

Add `src/lib/viewportAnchor.ts`:

- `getDashboardItemStableId(item)`:
  - Task: `itemId || contentId || id`
  - Group: `groupBlockId`
- `captureViewportAnchor({ dashboardItems, scrollTop, viewportHeight, rowHeight })`
  - Compute visible row range.
  - Build ordered candidates from the visible task rows, top to bottom.
  - Save each candidate's offset from viewport top.
  - Include group candidates only as fallback; the requirement prioritizes task items.
- `resolveViewportAnchor({ anchor, dashboardItems, rowHeight, maxScrollTop })`
  - Find the first captured task candidate still present.
  - Return `newIndex * rowHeight - capturedOffset`, clamped.
  - If no task candidate survives, use the first surviving group candidate.
  - If nothing survives, preserve the old scrollTop clamped to the new content height.

Add a small controller in dashboard context:

- `captureCurrentViewportAnchor()`
- `queueViewportRestore(anchor)`
- `consumePendingViewportAnchor()`

`ProjectDashboard` should register capture/restore behavior because it owns `sidebarRef` and `timelineRef` via `useScrollSync` (there is no `DashboardLayout` component). Since `useDashboardTasks` (where `setTasks` happens) has no access to the scroll refs, the capture function must be registered into the dashboard context by `ProjectDashboard` and invoked by the refresh path right before applying the snapshot.

Restore behavior:

- Run in `useLayoutEffect` after `dashboardItems` changes.
- Set both `sidebarRef.current.scrollTop` and `timelineRef.current.scrollTop` to the resolved value. Setting both directly is safe: `useScrollSync` handlers no-op when the two scrollTops already match within 0.5px.
- Preserve Gantt horizontal `scrollLeft`.
- No restoration should run for single item changes unless the changed task was inserted/removed and the operation intentionally queued an anchor.

### 7. Render Granularity Improvements

Phase 1 should rely on object identity preservation and existing `memo(SortableTaskRow)`.

After Phase 1, profile with React DevTools:

- If the sidebar is still expensive, check which props defeat `SortableTaskRow` memoization and stabilize callbacks/props where practical.
- If the Gantt is still visibly expensive, extract memoized `GanttTaskRow` and `GanttGroupRow` components keyed by stable IDs.
- If context-wide rerenders remain the bottleneck, introduce a no-dependency task store using `useSyncExternalStore`:
  - `useDashboardItemIds()`
  - `useTaskByStableId(id)`
  - `useTaskFields()`
  - Keep `DashboardContext` for commands and UI state.

Do not start with the external-store refactor unless Phase 1 fails acceptance criteria. It is a larger change and the current memoized row structure can benefit immediately from stable reconciliation.

## Acceptance Criteria

- `refresh_task` with a resolvable `itemId` or `contentId` does not call full project fetch.
- Background full refresh never replaces the task area with the initial loading spinner when tasks already exist.
- Full refresh preserves the first still-existing visible task item near the viewport top.
- If the top visible task was deleted, the next visible surviving task becomes the anchor.
- Redundant broadcasts from project/repo/owner channels do not trigger duplicate refreshes.
- Local reorder echoes continue to be skipped.
- Existing task selection remains if the selected task still exists; if it was removed, details/link state are cleaned up predictably.
- Unchanged task row objects keep reference identity across both single item refresh and full snapshot refresh.

## Risks and Edge Cases

- GitHub issue webhooks may provide only `contentId`, not the project item ID. The current local task list can resolve this only if the task is already present.
- GitHub API eventual consistency can return stale data shortly after a webhook. Keep the current webhook delay and retain local edit protection.
- Project item creation and reorder events may not include enough ordering context for a precise local patch. Use background full refresh with viewport preservation.
- A single dependency edit can legitimately change successor display dates, so "single element" does not always mean only one rendered row changes.
- Group-by-field changes can move a task between groups. Viewport anchoring must use stable task IDs, not row indexes.
- A refresh can move the anchor task into a collapsed group, removing it from `dashboardItems`. The "still present" check already handles this (the candidate simply isn't found and the next candidate is used), but a test should cover it.
- `issues`-sourced events never carry `projectId`, so per-project coalescing must key on the selected project, and missing-`projectId` events must not be dropped by project filtering.
- Background mode must not write `apiError` or `setTasks([])` on failure; otherwise a transient network error during a webhook refresh would blank the UI — the exact symptom this task is fixing.
