## Problem

1. The group-header drop plan (`getDashboardGroupDropPlan`) treated EVERY task-over-header
   drop as "insert into the group after its last child", regardless of drag direction. With a
   vertical sortable list, dragging an item upward onto a header previews the item landing
   ABOVE the header (the header shifts down) — so dragging a group's own task just above its
   group title and dropping it put the task back into the group (at the end), instead of
   moving it out.
2. The header highlight used dnd-kit's raw `isOver` (`isGroupDropActive = isTaskDropTarget &&
   isOver && !isDragging`), so it lit up even for the move-out case above, and it did NOT
   light up when a task hovered over another group's child rows even though that drop enters
   the group.

## Implementation

`src/lib/taskOrderUtils.ts`:
- `DashboardGroupDropPlan` gained `placement: 'into' | 'above'`
  (`getDashboardTaskGroupPathMovePlan` plans are tagged `'into'`).
- `getDashboardGroupDropPlan(items, activeSortId, overSortId, orderedTasks?)`:
  - New `isDropAboveGroupHeader` check: when the dragged task's visible index is greater than
    the header's index (upward drag), the simulated drop lands above the header. In that case
    the plan becomes `placement: 'above'` with `targetGroupPath = overGroup.path.slice(0, -1)`
    (the header's parent container — moving the task out of the group) and
    `afterTaskId = getAfterTaskIdForDropAboveGroup(...)`: the underlying-order predecessor of
    the group's first remaining task (same neighbor-anchoring approach as commit c083daf, so
    it stays correct under nested groups and "Group by Fields" re-sorting), with a visible
    `scanPreviousAnchor` fallback when no ordered task list is available or the group has no
    remaining children.
  - Downward drags onto a header, and all drops onto the synthetic project-root row (which is
    always at the top, used to ungroup), keep the previous "into the group, after its last
    child" behavior, now tagged `placement: 'into'`.
- New `getDashboardDropTargetGroupSortId(items, activeSortId, overSortId, orderedTasks?)`:
  resolves which group header should show the drop-target selection effect while a task drag
  hovers `overSortId`:
  - header hover with an `'into'` plan → that header; with an `'above'` plan → none;
  - task hover whose `getDashboardTaskGroupPathMovePlan` target is a non-empty group the
    dragged task is not already inside (`isPathPrefix` guard) → the containing group block,
    resolved by scanning upward from the hovered row so the correct block is picked even when
    the same path appears as multiple separated visible blocks;
  - same-group reorders, moves to the root, and group-block drags → none.

`src/components/Dashboard/TaskSidebar.tsx`:
- Tracks the current drag-over row via a new `onDragOver` handler (`dragOverSortId` state,
  cleared on drag start/end/cancel) and derives `dropTargetGroupSortId` with the new util
  (passing the context's underlying `tasks` order).
- `TaskGroupRow` now receives `isDropTargetGroup` and uses it for the highlight
  (`isGroupDropActive = isTaskDropTarget && isDropTargetGroup && !isDragging`) instead of the
  raw `useSortable().isOver`, so the selection effect follows the actual prospective drop.
- `handleDragEnd` passes `tasks` to `getDashboardGroupDropPlan` so the move-out anchor is
  resolved against the underlying project order. The drop itself still goes through
  `moveTaskToGroupPath(taskId, targetGroupPath, afterTaskId)`, which persists the path change
  and the reorder.

`docs/FEATURES.md`:
- Updated the Task Grouping feature description with the direction-aware header-drop and
  drop-target highlight behavior.

## Test patterns added (src/lib/taskOrderUtils.test.ts)

1. "plans moving a grouped task out of its group when dropped above the group header" — the
   reported bug case: first child dragged above its own header → `targetGroupPath: []`,
   `afterTaskId: null`, `placement: 'above'`.
2. "keeps a task outside the group when dragged from below onto the group header" — outside
   task dragged up onto a header lands above the group (`afterTaskId` = task above the group).
3. "moves a nested-group task to its parent group when dropped above the nested header" —
   nested header: target path becomes the parent group's path, anchored after the parent's
   preceding task.
4. "highlights the group header when a task hovers onto it from above" — into-drop highlight.
5. "highlights the target group when hovering a task that belongs to another group" —
   requirement 1: hovering a group's child row from outside highlights that group's header.
6. "does not highlight the group header when the drop would move the task above the group" —
   requirement 2: no selection effect for the move-out hover.
7. "does not highlight any group for same-group reorders".
8. "does not highlight when hovering an ungrouped task moves the task to the root".
Existing `getDashboardGroupDropPlan` / `getDashboardTaskGroupPathMovePlan` expectations were
extended with the new `placement` field; all previous behaviors are unchanged (downward header
drops, root-group ungroup drops, task-over-task group switches, ordering anchors).
