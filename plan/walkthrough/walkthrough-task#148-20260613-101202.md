# Task #148 Walkthrough

## Brief

Reduced task-list drag redraws by skipping duplicate drag-over state updates, avoiding repeated context-menu close writes during drag moves, and memoizing row callbacks/group rows; verified tests/lint/type-check/build and local mock dashboard smoke.

## Task

- Issue: #148
- Title: refine: drag and drop a task item in the task list view
- Request: When dragging and holding a task item in the task list view, the UI seemed to redraw at a high rate. Find the root cause and fix it.

## Implementation

- Investigated `TaskSidebar` drag state flow around `DndContext`.
- Identified avoidable render sources during drag:
  - `onDragOver` wrote `dragOverSortId` for every event, even when the hovered sortable id was unchanged.
  - `onDragMove` attempted to close the context menu every move even when it was already closed.
  - Task row memoization was weakened by unstable row action callbacks from `TaskSidebar`.
  - Group rows were not memoized while task rows already were.
- Updated `src/components/Dashboard/TaskSidebar.tsx`:
  - Added a `dragOverSortIdRef` guard so duplicate drag-over targets do not call React state setters.
  - Reset that ref on drag start, end, and cancel.
  - Guarded `setContextMenu(null)` in drag move so it only runs when a context menu is actually open.
  - Memoized `handleJumpToChart`, `handleJumpContextTargetToChart`, and `handleAddSuccessorsFromContext`.
- Updated `src/components/Dashboard/TaskSidebarRows.tsx`:
  - Wrapped `TaskGroupRow` in `memo`, matching `SortableTaskRow`.
- Updated `src/components/Dashboard/TaskSidebar.test.tsx`:
  - Captured mocked `DndContext` handlers.
  - Added a regression test proving duplicate drag-over targets do not rerender sortable rows.
- Updated `docs/FEATURES.md`:
  - Documented the drag-hover tracking and row memoization performance refinement.
