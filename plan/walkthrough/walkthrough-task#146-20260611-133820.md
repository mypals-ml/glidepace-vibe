# Walkthrough — Task #146: bug: change the order of the 2nd task to the 1st one in a group works wrongly

Date: 2026-06-11
Commit: c083daf "Fix top-of-group reorder in nested and field groups" (branch: develop)

## Brief execution walkthrough

Generalized drop-anchor resolution in `getVisibleDashboardMovePlan` so moving an item to the
beginning of a group anchors it directly above the group's original first task (resolved against
the underlying project order), and moving to the end keeps anchoring directly below the group's
last task. This fixes top-of-group drops jumping to the bottom of the group, including the
review-reported cases: groups nested under other path groups and `Group Path` groups shown
under "Group by Fields" re-sorting. Verified with 6 new unit test patterns plus the full
existing `taskOrderUtils` suite (36/36 passing), strict TypeScript checks, and ESLint on all
changed files.

## Problem

In the task list view, dragging the 2nd task of a group onto the 1st position made it jump to
the end of the group. A previous fix (e172f52) special-cased only the single group header
immediately above the drop position, so it still failed when:

- the group was a `Group Path` group nested under a "Group by Fields" group (two stacked
  headers above the first task), and generally in nested groups; and
- field grouping was active: `applyFieldGroupPaths` re-sorts the visible list by field key, so
  an `afterTaskId` anchored to the previous *visible* task (which belongs to a different field
  group) does not change the task's relative order inside its own group at all.

Reviewer-specified rules implemented:
- Moving item A to the BEGINNING of any group whose original first item is B places A as the
  closest upper neighbor of B.
- Moving item A to the END of any group whose original last item is C places A as the closest
  lower neighbor of C.

## Implementation

`src/lib/taskOrderUtils.ts`:
- Replaced `isGroupFirstChildInsertionBoundary` + `getPreviousAfterTaskId` with:
  - `scanPreviousAnchor` — walks upward from the simulated drop position. Task rows anchor
    "after that task". Collapsed group rows and the group header that was directly dropped onto
    represent their whole child block and anchor after their last non-moving child (preserves
    the existing "drop a group onto a group header places it after that group" behavior).
    Expanded group headers only mark the visible beginning of a group: the scan crosses them
    (reporting `crossedGroupStart`) and continues above — this alone fixes nested headers when
    visible order matches underlying order.
  - `getNextAnchorTaskId` — finds the first non-moving task below the drop position (the
    group's original first task B when dropping at a group beginning).
  - `getUnderlyingPredecessorTaskId` — resolves B's closest non-moving predecessor in the
    underlying ordered task list, so "place A after pred(B)" makes A the exact upper neighbor
    of B in the project order. This stays correct under "Group by Fields" re-sorting and under
    search filtering, because GitHub's position mutation and the local `moveTaskAfter` both
    operate on the underlying order while the stable field sort preserves relative order within
    each field group.
- `getVisibleDashboardMovePlan` and `getDashboardTaskGroupPathMovePlan` accept an optional
  `orderedTasks` parameter (the underlying full task order). When the backward scan crossed a
  group start and `orderedTasks` is provided, the plan anchors before the next task using the
  underlying order; otherwise it falls back to the (now header-skipping) backward scan, which
  remains correct when the visible order mirrors the underlying order.

`src/components/Dashboard/TaskSidebar.tsx`:
- `handleDragEnd` now passes the context's full `tasks` list to
  `getDashboardTaskGroupPathMovePlan` and `getVisibleDashboardMovePlan`.

`docs/FEATURES.md`:
- Updated the Task Grouping feature description to document the neighbor-anchoring rules.

## Test patterns added (src/lib/taskOrderUtils.test.ts)

1. "moves a task to the top of a path group nested under a field group" — two stacked headers
   (field group + path group); moving T2 above T1 must yield `afterTaskId: null`, not T3.
2. "moves a task to the top of a nested group right below its parent sibling task" — inner
   group beginning anchors to the parent's preceding task, with and without `orderedTasks`.
3. "anchors a top-of-group move to the underlying neighbor when field grouping re-sorts tasks"
   — underlying order `[B, A, X]` rendered as `k1:[X], k2:[B, A]`; moving A above B must anchor
   before B in the underlying order (`null`), not after the visible predecessor X.
4. "keeps same-group downward moves anchored to the visible previous task" — END-rule
   regression guard: A dropped after its same-group neighbor keeps `afterTaskId: 'A'`.
5. "moves a task to the end of its group as the closest lower neighbor of the last task" —
   END rule: dropping A onto last task C yields `afterTaskId: 'C'`.
6. "anchors moves after a collapsed group to that group last child" — collapsed rows still act
   as closed blocks (`afterTaskId: 'B'`).
