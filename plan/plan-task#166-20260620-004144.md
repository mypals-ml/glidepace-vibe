# Implementation and Test Plan: task#166 - bug: dismiss the task details view when switched to Forecast Dashboard

**Task from GitHub Issue #166 (mypals-ml/glidepace-vibe):**
- Title: bug: dismiss the task details view when switched to Forecast Dashboard
- Description: 
  bug: dismiss the task details view when switched to Forecast Dashboard
  - if the task details view is showing, when user click the Forecast Dashboard button, the details view should be dismissed

**Date:** 2026-06-20
**Current branch:** develop (pulled, up-to-date)
**Status field updated to In progress, comment added.**

## Detailed Implementation Plan

### Root Cause Analysis
- The Forecast button(s) are in:
  - `src/components/Dashboard/Views/DashboardViewSwitcher.tsx` (desktop header tabs)
  - `src/components/Header/HeaderOverflowMenu.tsx` (mobile overflow "More" menu)
- Both have `handleSwitch(view)` that:
  - For non-list: `setIsChartVisible(true); setDashboardView(view);`
  - No code touches `isTaskDetailsOpen` or `setIsTaskDetailsOpen`
- Task details rendering controlled in `src/components/ProjectDashboard.tsx`:
  - `shouldRenderTaskDetails = isTaskDetailsOpen && (isCreateMode || selectedTask !== null)`
  - Details panel rendered *unconditionally* (wrt dashboardView) when flag true, as sidebar pane or mobile overlay or desktop resizable aside.
- On forecast: sidebar hidden (`shouldShowTaskListPane` false for non-gantt desktop), main area uses `<ForecastDashboard />` (which ignores selectedTask / details state, uses only filteredTasks).
- Other close paths exist (mobile back nav hook, explicit close buttons, jump-to-gantt in sidebar/details, etc.) but not triggered by view switcher.
- Details can be opened only via gantt/sidebar interactions (TimelineChart clicks, TaskSidebar select/create), but the open flag persists across view switches.
- Result: details panel remains visible (or overlaying) when user switches to Forecast Dashboard.

### Changes Required
1. **Update view switch handlers (click paths) to dismiss details:**
   - Modify `handleSwitch` in `DashboardViewSwitcher.tsx`:
     - Destructure `setIsTaskDetailsOpen` (and `setIsCreateMode` for clean state) from useDashboard.
     - When `view === 'forecast'`: call `setIsTaskDetailsOpen(false); setIsCreateMode(false);` before/after setDashboardView.
   - Same for `handleSwitch` in `HeaderOverflowMenu.tsx` (already destructures some UI, add the sets).
   - This directly addresses "when user click the Forecast Dashboard button".

2. **Guard rendering in ProjectDashboard.tsx (defensive):**
   - Update `shouldRenderTaskDetails` to also require `dashboardView !== 'forecast'`.
   - This prevents any stale render of details panel (or create) when view is forecast, even if flags momentarily true.
   - Add a transition effect using ref to only dismiss on actual switch *to* forecast (optional but nice to avoid unnecessary sets).

3. **No changes needed to:**
   - ForecastDashboard.tsx (read-only)
   - TaskDetailsPanel / TaskDetailsContent (existing close logic)
   - Context / hooks (setters already available)
   - Mobile back nav (already handles dismiss)
   - selectedTaskId: leave as-is (selection state can persist; only view dismissed). If later needed, can clear but not per spec.
   - isCreateMode handling: we clear it on forecast switch to avoid stale create panel when returning to gantt.

4. **i18n / UI:** None. No new strings.
5. **Other files:** None (no API, no data model).
