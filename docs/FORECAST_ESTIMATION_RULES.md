# Forecast Estimation Rules

This document describes the exact rules used in the **Forecast Dashboard** (also referred to as the Estimated Completion / Burndown view) for calculating:

- The project's **estimated completion date**
- The **projected line** (future remaining workload)
- **Remaining workloads** used for forecasting

These rules were defined based on explicit product requirements and refined through implementation and design iteration (including the "A Stacked" variant).

---

## 1. Status-Based Remaining Workload Factors

Every task's effort is scaled by a **remaining workload factor** that depends on its current status. This factor represents how much of the task's estimated work is still considered "remaining" for forecasting purposes.

| Status          | Remaining Factor | Notes |
|-----------------|------------------|-------|
| Draft           | 0%               | Considered already "accounted for" or not yet real work |
| Todo            | 100%             | Full effort still remaining |
| In progress     | 50%              | Half the work is assumed done |
| In review       | 20%              | Mostly complete |
| Done            | 0%               | No remaining work |
| Any other       | 50%              | Default fallback |

**Effective effort for a task** = `estimateDays × factor`

Only non-Done tasks contribute to remaining work calculations.

---

## 2. Effective Remaining Effort

- `totalEstimateDays` = Sum of raw `estimateDays` across **all** tasks (used for chart scaling and overall effort display).
- For forecasting and allocation:
  - Only open (non-Done) tasks are considered.
  - `remainingDays` (used for projection) = sum of `(estimateDays × factor)` for all non-Done tasks.

---

## 3. Allocation of Work to Assignees

Remaining effort is allocated to individual people so that per-person finish dates can be calculated.

### Rules

1. **Assigned tasks**
   - Scale the effort first using the status factor.
   - Split the scaled effort equally among the assignees on that specific task:
     ```
     share = (estimateDays × factor) / number_of_assignees_on_task
     ```

2. **Unassigned tasks**
   - Scale using the factor.
   - Spread the scaled effort **fairly** across all currently active assignees in the project (equal share per person).

3. **No assignees at all in the project**
   - All remaining work is assigned to a single virtual "Unassigned" worker.
   - This also forces **Workers count = 1** in the UI.

Result: A map of `assigneeName → total allocated effective work (in person-days)`.

---

## 4. Per-Assignee Completion Date

For each assignee:

- Take their total allocated work (in effective days).
- Convert to whole workdays: `workDays = ceil(totalWork)`
- Advance the calendar from **today** by that many **workdays**.
- Workdays = Monday–Friday only (respecting the "5 days/week" capacity assumption).
- Result: `assigneeCompletion = today + addWorkdays(workDays)`

---

## 5. Project Estimated Completion Date

```
projectEstimatedCompletion = max( over all assignees' individual completion dates )
```

- This is the date shown prominently in the UI ("Estimated completion").
- It becomes the end of the x-axis range for the chart.
- If there is no remaining work → today.

---

## 6. Projected Remaining Workload & The Projected Line

The projected line (dashed) and future dots are generated by a forward simulation:

1. Start with the current effective `remainingDays`.
2. Walk forward one calendar day at a time (the full date range up to project completion).
3. On **workdays only**:
   - Every assignee who still has unspent personal allocation burns **up to 1** person-day.
   - Total `dayBurn` = sum of burns across all assignees that day.
4. `newRemaining = max(0, currentRemaining - dayBurn)`
5. Record the `remainingDays` value for that future date.

**Results**:
- The projected line starts from today's remaining value.
- It decreases according to available parallel capacity.
- It reaches **exactly 0** on the project estimated completion date.
- The slope is steepest when many assignees still have work; it flattens naturally as people finish their allocations.

Historical / actual points (solid line) continue to use task `targetDate` / `doneDate` logic and are **not** affected by the status factors.

---

## 7. Chart Date Range

- Start: Earliest relevant date among task dates + today.
- End: The capacity-based `projectEstimatedCompletion` (when remaining work > 0).
- If there is no remaining work, the range collapses to today.

---

## 8. Display of These Values (Assumptions & Visuals)

- **Task status remaining workload** section shows the fixed factors (0%/100%/50%/20%/0%/50%) for the known statuses.
- A stacked bar (when shown) visualizes the **proportional contribution** of each status to the current remaining work.
- Per-status rows usually show the factor % + the effective days for that status.
- **Workers count** falls back to 1 when there are no real assignees.
- Capacity is shown as "5d / week" (hard assumption).

---

## 9. Edge Cases & Fallbacks

- No assignees → 1 virtual worker.
- All work is already "Done" or has 0% factor → completion date = today, no projected segment.
- Tasks without explicit estimates → fall back to inclusive calendar duration between start/target.
- The simulation and completion calculations only consume capacity on workdays (Mon–Fri).
- Actual historical points still respect individual task target dates.

## Design Notes

These rules were developed to satisfy two goals simultaneously:
1. Produce a realistic **project-level** completion date even when individual task target dates are stale or in the past.
2. Make the **projected line** visually and numerically reflect available parallel capacity across assignees, using the status-based remaining workload percentages as the primary lever for "how much work is actually left."

The "A Stacked" visual treatment in the demo HTML was used to present the status workload factors clearly in the Assumptions area.
